# feat: WhatsApp Claude Bot Rewrite

## Overview

Rewrite `bot.sh` with a correct, long-term architecture based on the brainstorm decisions:
- Group-based membership (WhatsApp group, not self-chat)
- Prefix trigger (`!` or `@claude`) to eliminate reply loops architecturally
- SQLite `bot.db` for processed message IDs (replaces flat file)
- Self-filter: ignore bot's own JID always
- Stateless per-message Claude calls

Also package the setup process as a Claude Code skill (`/whatsapp-bot`).

## Current Problems

1. **Reply loop** — bot sends reply → reply lands in DB → bot processes reply → infinite loop
2. **Self-chat limitations** — all messages are `is_from_me=true`, no way to distinguish user from bot
3. **Flat file seen_ids** — grows forever, slow `grep` on large files
4. **Hardcoded paths** — `mise` Go path baked into scripts
5. **Two terminals required** — sync + bot must be manually managed; target users are non-technical

## Proposed Solution

### Architecture

```
WhatsApp Group "Claude Bot"
  → user sends "! explain recursion"
  → whatsapp-cli sync writes to store/messages.db
  → bot polls messages.db every 5s
  → SELECT messages WHERE group_jid = ? AND (content LIKE '!%' OR content LIKE '@claude%')
  → skip if sender JID == bot JID
  → skip if message ID already in bot.db processed_messages
  → strip prefix → run: claude --print "<prompt>"
  → whatsapp-cli send --to <group_jid> --message "<response>"
  → INSERT message ID into bot.db processed_messages
```

### Files to Create/Modify

```
bot.sh          — full rewrite (group JID, prefix filter, SQLite dedup)
sync.sh         — minor: use $PATH-resolved whatsapp-cli
config.sh       — new: GROUP_JID, BOT_JID, PREFIXES, POLL_INTERVAL
setup.sh        — new: interactive first-time setup wizard
store/bot.db    — new: SQLite DB for processed message IDs
```

## Implementation Phases

### Phase 1: Config & Setup Script

**`config.sh`** — sourced by bot.sh and sync.sh:
```bash
# store/config.sh (generated by setup.sh, gitignored)
GROUP_JID="120363XXXXXXXXX@g.us"
BOT_JID="919980527182@s.whatsapp.net"
PREFIXES=("!" "@claude")
POLL_INTERVAL=5
WA_BIN="$(which whatsapp-cli)"
CLAUDE_BIN="$(which claude)"
STORE_DIR="$(dirname "$0")/store"
```

**`setup.sh`** — interactive wizard (run once by the technical admin):
1. Check `whatsapp-cli` auth status; run auth if needed (shows QR code)
2. List available groups → numbered menu → admin picks the bot group
3. Prompt: "Enter your WhatsApp number with country code (e.g. 919980527182):"
4. Write `store/config.sh`
5. Initialize `store/bot.db` with schema
6. Start sync in background + bot in foreground automatically — no second terminal needed

**`bot.sh`** — starts sync as a background subprocess on launch:
```bash
# Start sync in background, kill it on exit
$WA_BIN sync &
SYNC_PID=$!
trap "kill $SYNC_PID 2>/dev/null" EXIT
```
This means the user (or admin) runs ONE command: `bash setup.sh` once, then `bash bot.sh` forever.

### Phase 2: Bot DB Schema

`store/bot.db`:
```sql
CREATE TABLE IF NOT EXISTS processed_messages (
    id TEXT PRIMARY KEY,
    processed_at INTEGER NOT NULL  -- epoch seconds
);
```

### Phase 3: Rewrite bot.sh

Key logic:

```bash
# Prefix check (strips prefix, returns clean prompt)
extract_prompt() {
  local content="$1"
  for prefix in "! " "!  " "@claude "; do
    if [[ "$content" == "$prefix"* ]]; then
      echo "${content#$prefix}"
      return 0
    fi
  done
  return 1  # not a bot command
}

# Query: only group messages with trigger prefix
SELECT id, sender, content FROM messages
WHERE chat_jid = '$GROUP_JID'
  AND sender != '$BOT_JID'
  AND (content LIKE '!%' OR content LIKE '@claude%')
ORDER BY timestamp ASC

# Dedup check via bot.db
SELECT 1 FROM processed_messages WHERE id = ?

# Mark processed
INSERT OR IGNORE INTO processed_messages (id, processed_at) VALUES (?, unixepoch())
```

### Phase 4: Claude Code Skill

Create `.claude/skills/whatsapp-bot.md`:

```markdown
# whatsapp-bot skill
Manages the WhatsApp Claude bot in ~/whatsapp-claudecode.

Commands:
- /whatsapp-bot setup    — run setup wizard
- /whatsapp-bot start    — run bot.sh (which manages sync internally)
- /whatsapp-bot stop     — stop all bot processes
- /whatsapp-bot status   — show running processes and last log lines
- /whatsapp-bot send     — send a message to the group (for Claude to use)
```

The `send` subcommand makes this **agent-native**: Claude can send WhatsApp messages as a tool action, not just respond to them.

## Acceptance Criteria

- [ ] Bot only responds to messages starting with `!` or `@claude` in the configured group
- [ ] Bot never responds to its own messages (sender JID filter)
- [ ] Same message ID is never processed twice (SQLite dedup)
- [ ] No reply loops under any circumstances
- [ ] `setup.sh` guides first-time config interactively
- [ ] `store/config.sh` holds all environment-specific values (no hardcoded paths)
- [ ] `/whatsapp-bot` skill works for setup, start, stop, status
- [ ] Bot survives sync restarts without re-processing old messages
- [ ] Single command to start everything (`bash bot.sh`) — no second terminal needed
- [ ] `setup.sh` is the only setup step an admin needs to run

## Dependencies & Risks

| Risk | Mitigation |
|---|---|
| Group JID not stable | Detect from `chats list` at setup time, store in config |
| `whatsapp-cli` path varies | Resolve via `which` at setup time |
| Sync DB lock still possible | Direct SQLite read with `?mode=ro` (already proven to work) |
| Claude `--print` flag behavior | Already tested: `env -u CLAUDECODE claude --print` works |
